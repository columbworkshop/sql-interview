# Linux Security

## Что такое seccomp и какую главную проблему безопасности он решает?

seccomp (Secure Computing) — это механизм безопасности ядра Linux, который работает как файрвол для системных вызовов (syscalls).
Когда приложение хочет сделать что-то за пределами своей памяти (открыть файл, запустить процесс, открыть сетевой порт), оно обращается к ядру через системные вызовы. 
В Linux их более 300, но обычному веб-серверу нужно от силы 30-50.
Главная задача seccomp — уменьшение поверхности атаки. 
seccomp не даст выполнить вредоносные системные вызовы (например, execve для запуска шелла или ptrace для чтения памяти других процессов).

## Какие два основных режима работы есть у seccomp? В чем их разница?

1. Strict mode (seccomp=1): Самый жесткий и старый режим. Разрешает процессу выполнять только 4 вызова: read(), write(), _exit() и sigreturn(). Любой другой вызов мгновенно убивает процесс сигналом SIGKILL. Полезен для очень изолированных математических песочниц, но неприменим для обычного софта.
2. Filter mode (seccomp=2 / seccomp-bpf): Современный и гибкий режим. Он использует BPF (Berkeley Packet Filter) для написания правил. Можно создать профиль (обычно это JSON-файл), в котором описываются белые (allowlist) или черные (blocklist) списки системных вызовов, а также фильтрация по их аргументам.

## В чем принципиальное отличие seccomp от других систем безопасности Linux — AppArmor и SELinux? Нужно ли использовать их вместе?

Разница в уровне абстракции:

1. seccomp работает на самом низком уровне — уровне взаимодействия с ядром. Он запрещает саму "функцию" (например, "вообще запретить вызов mount"). Ему всё равно, какие файлы вы трогаете, он фильтрует глаголы.
2. AppArmor и SELinux — это системы мандатного управления доступом (MAC). Они работают на уровне объектов файловой системы и процессов (например, "разрешить вызов open, но только для чтения из /etc/nginx/, а запись в /etc/passwd запретить").
3. Нужно ли вместе? Да. Это паттерн "Эшелонированная защита" (Defense in Depth). seccomp отсекает опасные возможности ядра, а AppArmor защищает файловую систему и сеть.

## Как Docker и Kubernetes по умолчанию относятся к seccomp?

Docker по умолчанию применяет к каждому контейнеру базовый профиль seccomp (Default Profile), который блокирует около 44 потенциально опасных системных вызовов из 300+ (например, блокирует reboot, манипуляции с часами, модулями ядра и BPF).
Kubernetes до версии 1.22 работал в режиме Unconfined (seccomp был выключен для подов). Однако, начиная с K8s 1.22 (и окончательно став стандартом в 1.27), поды по умолчанию используют профиль RuntimeDefault (делегируя настройки рантайму, например containerd), если разработчик явно не указал Unconfined.

## Приложение падает с ошибкой (или просто тихо умирает) после того, как к поду применили жесткий seccomp-профиль. Как ты будешь диагностировать, какой именно системный вызов заблокировал seccomp?

Есть 3 основных пути решения этой проблемы:

1. Посмотреть Audit Logs (самый правильный путь на проде): Когда seccomp убивает процесс, ядро обычно пишет это в лог аудита узла. Нужно зайти на Node (рабочий узел) и посмотреть /var/log/audit/audit.log (или dmesg), поискав записи type=SECCOMP или type=ANOM_ABEND. Там будет указан номер заблокированного syscall'а.
2. Прогнать через strace (путь локальной отладки): Запустить приложение без seccomp-профиля с помощью команды strace -c ./my-app. Она соберет статистику абсолютно всех системных вызовов, которые делает приложение за время работы. Полученный список нужно сравнить с разрешенными в JSON-профиле.
3. Режим аудита в профиле: Временно изменить действие по умолчанию в seccomp-профиле (JSON) с SCMP_ACT_ERRNO (блокировка) на SCMP_ACT_LOG. Тогда seccomp не будет убивать приложение, а просто запишет в syslog все "незаконные" вызовы.

## Какие capabilities есть и какие наиболее опасные?

CAP_SYS_ADMIN - фактически это "новый root". позволяет монтировать файловые системы (mount), вызывать bpf() (работать с eBPF), управлять неймспейсами (namespaces), изменять лимиты дисковых квот, работать с ptrace (в старых ядрах) и делать еще десятки критических вещей.
CAP_SYS_MODULE - позволяет загружать и выгружать модули ядра Linux (через init_module и finit_module).
CAP_SYS_PTRACE - позволяет использовать системный вызов ptrace() для наблюдения и контроля за другими процессами.
CAP_SYS_BOOT - позволяет вызывать системный вызов reboot(), а также загружать новые ядра через kexec_load().

Docker по умолчанию: Отбирает (drop) все capabilities и выдает обратно только базовый, относительно безопасный набор (около 14 штук: CAP_CHOWN, CAP_DAC_OVERRIDE, CAP_FOWNER, CAP_FSETID, CAP_KILL, CAP_SETGID, CAP_SETUID, CAP_SETPCAP, CAP_NET_BIND_SERVICE, CAP_NET_RAW, CAP_SYS_CHROOT, CAP_MKNOD, CAP_AUDIT_WRITE, CAP_SETFCAP).CAP_SYS_ADMIN и CAP_SYS_MODULE в этом списке нет.
Kubernetes Security Context: В K8s хорошей практикой считается сбросить все возможности и добавить только нужные

```yaml
securityContext:
  capabilities:
    drop:
      - ALL
    add:
      - NET_BIND_SERVICE
```
